# Think Before Coding

**Don't assume. Don't hide confusion. Surface tradeoffs.**

Before implementing:

- State your assumptions explicitly. If uncertain, ask.
- If multiple interpretations exist, present them - don't pick silently.
- If a simpler approach exists, say so. Push back when warranted.
- If something is unclear, stop. Name what's confusing. Ask.

# Goal-Driven Execution

**Define success criteria. Loop until verified.**

Transform tasks into verifiable goals:

- "Add validation" → "Write tests for invalid inputs, then make them pass"
- "Fix the bug" → "Write a test that reproduces it, then make it pass"
- "Refactor X" → "Ensure tests pass before and after"

For multi-step tasks, state a brief plan:

```
1. [Step] → verify: [check]
2. [Step] → verify: [check]
3. [Step] → verify: [check]
```

Strong success criteria let you loop independently. Weak criteria ("make it work") require constant clarification.

# Monorepo Structure

This directory is a Git monorepo containing a `pnpm` workspace:

- The codebase is primarily TypeScript.
- All packages live in `packages/`.
- The core OpenCLI package is `packages/opencli`.

In error stack traces, built files from workspace packages in `node_modules/` map to TypeScript source in `packages/`:

- `node_modules/astro/dist/...` → `packages/astro/src/...`

Edits to source files take effect after rebuilding the package via `pnpm build`.

Use `pnpm -C <dir> <command>` for project-local script commands when working in packages/examples/triage directories (Example: `pnpm -C packages/opencli build`, `pnpm -C examples/aichat dev`). Only omit `-C` flag when intentionally working in the monorepo root (Example: `pnpm format`, `pnpm lint`).

# Running tasks

```shell
pnpm run build              # build all packages
pnpm run build:affected     # build only changed packages
pnpm run test               # full test suite
pnpm run test:affected      # tests for changed packages only
pnpm run lint               # lint (does not auto-fix)
pnpm run format             # auto-format with Biome + Prettier
```

# Commits

Angular Commit Convention, enforced by `commitlint`. Violations are rejected by the pre-commit hook.

```
<type>(<scope>): <short description>
```

- Subject starts with a **lowercase** letter, no trailing period, under 100 characters.
- Allowed types: `build` `chore` `ci` `docs` `examples` `feat` `fix` `perf` `refactor` `revert` `style` `test` `types`

```
feat(cli): add --verbose flag
fix(parser): handle empty input
test(cli): cover edge cases in argument parsing
```

# Changesets

Every PR that touches a publishable package needs a changeset:

```bash
pnpm changeset
# select packages → patch / minor / major → write a summary
```

Commit the generated file in `.changeset/` alongside your code. Skip for CI-only, docs-only, or internal script changes.

# Adding a new package

**Do not create package directories manually.** Always use the generator:

```bash
pnpm create-package <name>
```

It scaffolds `src/index.ts`, `__tests__/.gitkeep`, all config files, and updates `.github/labels.yml`, `labeler.yml`, and `issue_labeler.yml`. A manually created package will be missing all of that.

After scaffolding:

```bash
pnpm install && pnpm run build
```

# Things not to do

**Do not edit `.changeset/` by hand.** Those files are generated by `pnpm changeset`. Manual edits corrupt the versioning state.

**Do not edit `pnpm-lock.yaml` directly.** It is generated by `pnpm install`. To update a dependency, edit the relevant `package.json` and run `pnpm install`.

**Do not bump versions or edit `CHANGELOG.md`.** Releases are fully automated via Changesets. Manual edits conflict with the release PR.
